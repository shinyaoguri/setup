# グローバル CLAUDE.md

全プロジェクト共通の個人規約。実体は setup リポジトリの `claude/CLAUDE.md`
(settings.json とあわせて tasks/claude.yml が `~/.claude/` 配下へシンボリックリンク)。
プロジェクト・リポジトリ固有の規約は各所の CLAUDE.md / CONTRIBUTING.md が正本で、
ここでは繰り返さない。

## 全作業共通

- 応答・説明は日本語 (settings.json の `language` が主、ここは確認用)。成果物の言語は用途と指示に従う
- 品質を最優先し、既定は最高性能のモデルで走らせる。明らかに単純で結果が変わらない作業 (定型的な整形・変換・調査など) に限り、コストの安いモデルのサブエージェントへ委譲する。迷ったら高性能側に倒す
- やり直しがきく作業は、経過と根拠を残しながら指示を待たず進めてよい。不可逆な操作・外部へ影響が及ぶ操作 (送信・公開・共有・削除・履歴の書き換えなど) だけは事前にユーザーへ確認する
- セッションの記憶 (auto-memory 含む) はいつでもリセットされる前提。決定・発見・方針変更に加え、進行中の状況・課題・残タスク・調査の中間所見・検証ログも、その作業の恒久的な置き場 (開発なら Issue・PR とそのコメント・docs、それ以外なら該当ドキュメント等) へ積極的に記録する。複数ステップの作業は着手時に計画を宣言し、長い調査は中間結果をこまめに追記して、記憶ゼロからでも置き場を読むだけで状況把握・再開できる状態を保つ
- 問題・ミス・手戻りが起きたら、応急処置で終わらせず、同じ問題を繰り返さないための改善策もあわせて検討・提案する
- 改善策の手段は CLAUDE.md への追記に限らず、hooks・スクリプト・CI・利用ソフトやサービスの設定まで多角的に比較し、根本原因に効く最もシンプルな一手を選ぶ。決定論的に強制できる仕組み (hooks・lint・CI 等) を文書ルールより優先する
- 個別の対症ルールを積み増して全体を肥大させない (個別最適より全体最適)。置き場は問題のスコープに合わせる (プロジェクト固有はそのリポジトリへ、全プロジェクト共通のみここへ)
- 秘密情報・個人情報・実データは、成果物・ログ・外部サービスへ不用意に持ち出さない
- スクリーンショット等の画像読み込みはトークン高コスト。検証はまずテキストベース (DOM・状態・ログの読み取り) で進め、視覚チェックは人間へ依頼する。画像を読むのは他の手段で理解できないときに限る (完成度は妥協しない)

## ソフトウェア開発

- コードの識別子は英語。コメント・ドキュメント・コミット要約は日本語でよい
- GitHub Flow: main が唯一の長命ブランチ。作業は main から `<type>/<短い説明>` ブランチを切り、小さく作って早めに PR (WIP なら Draft)
- コミットは Conventional Commits: `<type>(<scope>): <要約>`。type は feat / fix / docs / refactor / test / chore / ci。1 コミット 1 関心
- Squash merge 前提。PR タイトルもコミットと同形式に書く (マージコミットのメッセージになる)
- コミットログと PR 本文 (目的・変更点・確認方法) を丁寧に書くこと。それが書けていて CI が green なら、push / merge は指示を待たず進めてよい (force push・main への直接 push は上記の不可逆操作として事前確認)
- merge 後は main に戻って pull し、`git fetch -p` でローカルブランチを掃除
- 1 PR = 1 関心事。本題以外のバグ・改善に気付いたら直接修正せず、`gh issue list --search` で重複を軽く見てから `gh issue create` で起票
- 仕様・設計 docs は叩き台として扱い、曖昧点・軽微な矛盾は同一 PR で docs も改訂して根拠を残す。確定要件・設計判断級の変更は事前にユーザーへ確認する
- テストは実装 PR に含める: バグ修正は失敗する再現テストが先 / 正常系に加えて失敗系・境界値を最低一つ / 新しいテストは検証対象の振る舞いを一時的に壊して赤くなるのを見てから仕上げる (壊しても緑のままのテストは書き直す)
- 秘密情報の置き場は gitignore 済みの `.env` などに限る (コード・fixtures・ログ・コミットに入れない)
- Claude Code のスキルは用途で置き場を分ける: 複数プロジェクトで使う汎用スキルは setup の `claude/skills/` (symlink で全プロジェクトへ供給)、特定プロジェクト固有のものは各リポジトリの `.claude/skills/` へ。第三者配布スキルはコピーせず plugin marketplace 経由で入れる (settings.json で宣言し上流更新に追従)
