# グローバル CLAUDE.md

全プロジェクト共通の個人規約。実体は setup リポジトリの `claude/CLAUDE.md`
(tasks/claude.yml が `~/.claude/CLAUDE.md` へシンボリックリンク)。
プロジェクト固有の規約(スタック・コマンド・アーキテクチャ・絶対規則)は
各リポジトリの CLAUDE.md / CONTRIBUTING.md が正本で、ここでは繰り返さない。

## 言語

- 応答・説明は日本語 (settings.json の `language` が主、ここは確認用)
- コードの識別子は英語。コメント・ドキュメント・コミット要約は日本語でよい

## Git / GitHub (GitHub Flow)

- main が唯一の長命ブランチ。作業は main から `<type>/<短い説明>` ブランチを切り、小さく作って早めに PR (WIP なら Draft)
- コミットは Conventional Commits: `<type>(<scope>): <要約>`。type は feat / fix / docs / refactor / test / chore / ci。1 コミット 1 関心
- Squash merge 前提。PR タイトルもコミットと同形式に書く (マージコミットのメッセージになる)
- コミットログと PR 本文 (目的・変更点・確認方法) を丁寧に書くこと。それが書けていて CI が green なら、push / merge は指示を待たず積極的に進めてよい
- force push・main への直接 push・履歴の書き換えなど不可逆な操作だけは、事前にユーザーへ確認する
- merge 後は main に戻って pull し、`git fetch -p` でローカルブランチを掃除
- 1 PR = 1 関心事。本題以外のバグ・改善に気付いたら直接修正せず、`gh issue list --search` で重複を軽く見てから `gh issue create` で起票

## 情報の残し方

- セッションのメモリ・チャット履歴は揮発する前提。重要な決定・発見・方針変更は Issue コメント・docs・ADR・PR に残し、記憶ゼロからでも再開できる状態を保つ
- 仕様・設計 docs は叩き台として扱い、曖昧点・軽微な矛盾は同一 PR で docs も改訂して根拠を残す。確定要件・設計判断級の変更は事前にユーザーへ確認する

## テスト

- 機能実装の PR には、その動作を検証するテストを含める
- バグ修正は、まず失敗する再現テストを書いてから直す
- 正常系だけでなく、失敗系・境界値を最低一つ入れる
- 新しいテストは、実装を一時的に壊して赤くなることを見てから仕上げる (壊しても緑のままのテストは書き直す)

## 秘密情報

- API キー等の秘密情報・実データは、コード・fixtures・ログ・コミットに入れない。置き場は gitignore 済みの `.env` などに限る
